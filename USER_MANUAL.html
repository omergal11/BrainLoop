1. Introduction & Installation

1.1 What is BrainLoop?

BrainLoop is an interactive learning system for practicing Computer Science topics (e.g., Data Structures, Algorithms, Databases, Operating Systems).
It provides:

  • Quiz sessions (code questions and multiple‑choice questions)
  • Detailed per‑topic feedback and session summaries
  • Personal statistics for each learner
  • Aggregated analytics for administrators (e.g., top students, most challenging topics)

1.2 System Requirements

Operating System

  • Windows 10/11, macOS, or modern Linux

Software

  • Docker: Latest version installed
  • Docker Compose: Latest version installed
  • Browser: modern browser (Chrome, Firefox, Edge)

📋 Getting Started with Docker

Step 1 – Start All Services

Open a terminal/command prompt in the BrainLoop project folder and run:

docker-compose -f docker-compose.dev.yml up --build

Docker automatically handles all setup!

Step 2 – Access Your Apps

After startup completes (you'll see "Application startup complete" in the logs):

  • Frontend App: http://localhost:5173
  • API Documentation: http://localhost:8000/docs
  • Health Check: http://localhost:8000/health

Step 3 – Stop Services

Press Ctrl+C in the terminal, or run:

docker-compose -f docker-compose.dev.yml down

🌐 Accessing the Application

Service URL
Frontend (App) http://localhost:5173
Backend (API) http://localhost:8000

✅ If the backend is running correctly, visiting http://localhost:8000/ will return:

{"message": "BrainLoop API is running"}

BrainLoop supports two main roles:

  • Standard User (Student) – solves questions, tracks their own progress.
  • Admin – has access to system-wide analytics and (optionally) content management.

2.2 Registration Rules

Users can also self-register via the Sign Up tab on the Auth page:

  • Username
    • Only English letters and digits (no spaces, no special characters)
  • Email
    • Must be a valid email format
  • Password
    • Minimum 8 characters
    • At least one uppercase letter (A-Z)
    • At least one lowercase letter (a-z)
    • At least one digit (0-9)
    • At least one special character: @$!%*?&#
  • Birthdate
    • Valid past date (not in the future)

If any field is invalid, inline validation messages are shown and the registration cannot proceed.

2.3 Authentication & Tokens

  • After login, the backend returns a JWT access token.
  • The frontend stores this token and user metadata in localStorage:
    • brainloop_token – JWT access token
    • brainloop_user_id – User ID
    • brainloop_is_admin – Admin flag (true/false)
  • All protected API calls include the token in the Authorization: Bearer <token> header.

Token expiry

  • Tokens expire after 30 minutes (configurable via backend environment variable).
  • When a token is expired or invalid, API calls fail and the frontend shows a "token expired" modal.
  • The user is then asked to log in again to obtain a fresh token.

Logout

  • Clicking Logout in the header clears all BrainLoop entries from localStorage and navigates to the Auth page (/auth).

4. Database Structure

BrainLoop uses a MySQL relational database. The actual schema (from brainloop_db.sql) is:

4.1 Core Tables

  • Users

    • user_id (PK)
    • username (unique)
    • password
    • email
    • birth_date
    • is_admin (0 = student, 1 = admin)

  • Topics

    • topic_id (PK)
    • name (unique)

  • Questions

    • q_id (PK)
    • question_text
    • type (e.g.┬ácode, choose)
    • option_a, option_b, option_c, option_d
    • correct_answer
    • topic_id (FK → Topics.topic_id)

  • User_Answers

    • answer_id (PK)
    • user_id (FK → Users.user_id)
    • q_id (FK → Questions.q_id)
    • user_answer
    • is_correct (0/1)

  • Learning_Sessions

    • session_id (PK)
    • user_id (FK → Users.user_id)
    • end_time
    • questions_solved (number of correct answers)

  • User_Strikes

    • user_id (PK, FK → Users.user_id)
    • current_streak_days
    • longest_streak
    • current_streak_start
    • last_activity_date

Note: Earlier draft names like answers, user_tests, scores do not exist in the final schema and were removed here to match the actual database.

4.2 Key Relationships

  • Questions.topic_id → Topics.topic_id
  • User_Answers.q_id → Questions.q_id
  • User_Answers.user_id → Users.user_id
  • Learning_Sessions.user_id → Users.user_id
  • User_Strikes.user_id → Users.user_id

This design supports:

  • Per-user and per-topic analytics
  • Session-level tracking
  • Daily streaks and admin reporting

5. Complex SQL Queries

The backend uses raw SQL (no ORM). The most complex queries live in backend/routes/stats.py and implement the statistics features.

5.1 User Statistics (Personal Dashboard)

These queries power /stats/user and the Statistics page.

  • Overall success rate

    • Counts all rows in User_Answers for the current user.
    • Sums only rows where is_correct = 1.
    • Computes success_rate = 100 * correct / total.
    SELECT 
    COUNT(*) AS total_questions,
    SUM(CASE WHEN is_correct = 1 THEN 1 ELSE 0 END) AS correct_answers
FROM User_Answers
WHERE user_id = %s;

  • Strongest topic

    • Joins User_Answers → Questions → Topics.
    • Groups by topic name.
    • For each topic:
      • total attempts
      • correct answers
      • accuracy in percent.
    • Orders by accuracy DESC, total DESC.
    • Returns the top row.
    SELECT 
    t.name,
    COUNT(*) AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(*),
        2
    ) AS accuracy
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
JOIN Topics   t ON q.topic_id = t.topic_id
WHERE ua.user_id = %s
GROUP BY t.name
HAVING COUNT(*) > 0
ORDER BY accuracy DESC, COUNT(*) DESC
LIMIT 1;

  • Weakest topic

    • Same join and grouping as above.
    • Orders by accuracy ASC, total DESC.
    • Returns the lowest-accuracy topic.
    SELECT 
    t.name,
    COUNT(*) AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(*),
        2
    ) AS accuracy
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
JOIN Topics   t ON q.topic_id = t.topic_id
WHERE ua.user_id = %s
GROUP BY t.name
HAVING COUNT(*) > 0
ORDER BY accuracy ASC, COUNT(*) DESC
LIMIT 1;

  • Last completed session

    • From Learning_Sessions:
      • Filters by user_id and end_time IS NOT NULL.
      • Orders by session_id DESC.
      • Takes the last row.
    SELECT 
    session_id,
    start_time,
    end_time,
    questions_solved
FROM Learning_Sessions
WHERE user_id = %s
  AND end_time IS NOT NULL
ORDER BY session_id DESC
LIMIT 1;

  • Best session

    • From Learning_Sessions:
      • Filters by user_id, end_time IS NOT NULL, questions_solved > 0.
      • Orders by:
        • questions_solved DESC (most questions solved)
        • duration ASC (fastest)
        • session_id DESC (most recent).
      • Returns the "best" single session.
    SELECT 
    session_id,
    start_time,
    end_time,
    questions_solved
FROM Learning_Sessions
WHERE user_id = %s
  AND end_time IS NOT NULL
  AND questions_solved > 0
ORDER BY
    questions_solved DESC,
    (UNIX_TIMESTAMP(end_time) - UNIX_TIMESTAMP(start_time)) ASC,
    session_id DESC
LIMIT 1;

  • Per-topic breakdown

    • Joins User_Answers → Questions → Topics.
    • Groups by topic name.
    • For each topic:
      • Stores total, correct, and derived accuracy.
    • The Statistics page uses this to display the "All Topics" grid.
    SELECT 
    t.name AS topic,
    COUNT(*) AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
JOIN Topics   t ON q.topic_id = t.topic_id
WHERE ua.user_id = %s
GROUP BY t.name;

5.2 Admin Statistics (Global Analytics)

These queries power /admin/stats/overview and the Admin Insights page.

  • Top performing students

    • Users LEFT JOIN User_Answers.
    • For each user:
      • total_questions, correct_answers, success_rate.
    • Filters to users with at least one answer.
    • Orders by success_rate DESC.
    • Returns top N (e.g., 10).
    SELECT 
    u.user_id,
    u.username,
    COUNT(ua.answer_id) AS total_questions,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct_answers,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(ua.answer_id),
        2
    ) AS success_rate
FROM Users u
LEFT JOIN User_Answers ua ON u.user_id = ua.user_id
GROUP BY u.user_id, u.username
HAVING COUNT(ua.answer_id) > 0
ORDER BY success_rate DESC
LIMIT 10;

  • Most practiced topics

    • User_Answers JOIN Questions LEFT JOIN Topics.
    • Groups by topic.
    • Counts answer_count per topic.
    • Orders by answer_count DESC.
    • Returns top N (e.g., 5).
    SELECT 
    COALESCE(t.name, 'Unknown') AS topic_name,
    COUNT(ua.answer_id)        AS answer_count
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
LEFT JOIN Topics t ON q.topic_id = t.topic_id
GROUP BY t.name
ORDER BY answer_count DESC
LIMIT 5;

  • Most challenging topics

    • Same join as above.
    • Computes:
      • total
      • correct
      • success_rate = correct / total * 100.
    • Requires a minimum number of attempts (e.g., HAVING COUNT(...) >= 3) to reduce noise.
    • Orders by success_rate ASC.
    SELECT 
    COALESCE(t.name, 'Unknown') AS topic_name,
    COUNT(ua.answer_id)         AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(ua.answer_id),
        2
    ) AS success_rate
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
LEFT JOIN Topics t ON q.topic_id = t.topic_id
GROUP BY t.name
HAVING COUNT(ua.answer_id) >= 3
ORDER BY success_rate ASC
LIMIT 5;

  • Topics with highest proficiency

    • Same aggregation as "most challenging".
    • Orders by success_rate DESC.
    SELECT 
    COALESCE(t.name, 'Unknown') AS topic_name,
    COUNT(ua.answer_id)         AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(ua.answer_id),
        2
    ) AS success_rate
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
LEFT JOIN Topics t ON q.topic_id = t.topic_id
GROUP BY t.name
HAVING COUNT(ua.answer_id) >= 3
ORDER BY success_rate DESC
LIMIT 5;

  • Average session duration

    • From Learning_Sessions where end_time IS NOT NULL:
      • Uses AVG((end_time - start_time) / 60) (in minutes).
    SELECT 
    AVG(
        (UNIX_TIMESTAMP(end_time) - UNIX_TIMESTAMP(start_time)) / 60.0
    ) AS avg_minutes
FROM Learning_Sessions
WHERE end_time IS NOT NULL;

  • Strongest/weakest topic overall

    • Computes a global success_rate per topic using answers across all users.
    • Orders descending for strongest, ascending for weakest.
    • Returns one topic in each query.

    – Strongest overall topic

    SELECT 
    COALESCE(t.name, 'Unknown') AS topic_name,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(ua.answer_id),
        2
    ) AS success_rate
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
LEFT JOIN Topics t ON q.topic_id = t.topic_id
GROUP BY t.name
HAVING COUNT(ua.answer_id) > 0
ORDER BY success_rate DESC
LIMIT 1;

    – Weakest overall topic

    SELECT 
    COALESCE(t.name, 'Unknown') AS topic_name,
    ROUND(
        CAST(SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS FLOAT)
        * 100.0 / COUNT(ua.answer_id),
        2
    ) AS success_rate
FROM User_Answers ua
JOIN Questions q ON ua.q_id = q.q_id
LEFT JOIN Topics t ON q.topic_id = t.topic_id
GROUP BY t.name
HAVING COUNT(ua.answer_id) > 0
ORDER BY success_rate ASC
LIMIT 1;

  • Streak leaders & best overall session

    • From User_Strikes:
      • User with the largest longest_streak.
      • User with the largest current_streak_days.
    • From Learning_Sessions + Users:
      • Session with most questions_solved and shortest duration.

    – Longest streak of all time

    SELECT 
    u.user_id,
    u.username,
    us.longest_streak
FROM Users u
JOIN User_Strikes us ON u.user_id = us.user_id
WHERE us.longest_streak > 0
ORDER BY us.longest_streak DESC
LIMIT 1;

    – Current longest streak

    SELECT 
    u.user_id,
    u.username,
    us.current_streak_days
FROM Users u
JOIN User_Strikes us ON u.user_id = us.user_id
WHERE us.current_streak_days > 0
ORDER BY us.current_streak_days DESC
LIMIT 1;

    – Best session overall

    SELECT 
    ls.session_id,
    u.username,
    ls.start_time,
    ls.end_time,
    ls.questions_solved,
    u.user_id,
    ROUND(
        (UNIX_TIMESTAMP(ls.end_time) - UNIX_TIMESTAMP(ls.start_time)) / 60.0,
        2
    ) AS duration_minutes
FROM Learning_Sessions ls
JOIN Users u ON ls.user_id = u.user_id
WHERE ls.end_time IS NOT NULL
  AND ls.questions_solved > 0
ORDER BY 
    ls.questions_solved DESC,
    (UNIX_TIMESTAMP(ls.end_time) - UNIX_TIMESTAMP(ls.start_time)) ASC
LIMIT 1;

5.3 Question Selection (Random Batches)

Used for building quizzes.

  • When topics are selected:

    • Builds a dynamic IN ( ... ) list with placeholders.

    • E.g.:

      SELECT * FROM Questions
WHERE type = ?
  AND topic_id IN (?, ?, ...)
ORDER BY RAND()
LIMIT ?
  • When no topics are selected:

    • Selects by type only:

      SELECT * FROM Questions
WHERE type = ?
ORDER BY RAND()
LIMIT ?

This ensures each quiz is random but respects the filters chosen on the Home page.

6. Analytics Pages (Statistics & Admin Insights)

6.1 Personal Analytics: "Statistics" Page (/stats)

The Statistics page shows a logged-in user a summary of their overall learning performance.

Access

  • Only when logged in.
  • Open via the "Statistics" icon in the header.

Data refresh

  • Automatically loads the latest stats on page load.
  • A "Refresh" button lets the user reload manually.
  • When the browser tab becomes visible again (e.g., after finishing a quiz and coming back), the page may auto-refresh.

Overall performance

  • Total Questions – number of answers submitted.
  • Correct Answers – number of correct answers.
  • Success Rate – correct / total * 100 (rounded).
  • Longest Streak – the highest number of consecutive practice days ever achieved (from User_Strikes).

Sessions

  • Last Session
    • Date of last completed session.
    • Number of questions answered correctly in that session.
  • Best Session
    • Session with highest questions_solved, breaking ties by shortest duration.
    • Shows date and questions solved.

Topic performance

  • Strongest Topic
    • Topic with the highest accuracy (based on User_Answers).
  • Weakest Topic
    • Topic with the lowest accuracy (with enough attempts).
  • All Topics section
    • Per-topic cards showing:
      • Topic name
      • Progress bar 0–30–60–100%
      • "X of Y correct"

Navigation

  • Back to Home button returns to the main practice selection screen.

6.2 Admin Analytics: "Admin Insights" Page (/admin/stats)

The Admin Insights page shows system-wide analytics, based on all users' activity.

Access

  • Only for admins (is_admin = true).
  • Non-admin users are redirected to /home.

Summary metrics

  • Average Session Duration
    • Average time (in minutes) for completed sessions.
  • Strongest Topic (Overall)
    • Topic with highest global success rate.
  • Weakest Topic (Overall)
    • Topic with lowest global success rate (with enough attempts).
  • Longest Streak (All Time)
    • User with the greatest longest_streak and number of days.
  • Current Longest Streak
    • User currently holding the longest active streak.
  • Best Session Overall
    • The single most impressive session:
      • User
      • Questions solved
      • Duration (minutes + seconds)

Leaderboards & topic lists

  • Top Performing Students
    • Based on success rate and total questions answered.
  • Most Frequently Practiced Topics
    • Topics with highest answer counts.
  • Most Challenging Topics
    • Topics with lowest success rate and enough attempts.
  • Topics with Highest Proficiency
    • Topics with highest success rate and sufficient attempts.

Navigation

  • A Back to Home button returns to the main screen.
